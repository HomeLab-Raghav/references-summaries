CH · 01

What Is Wireshark & How It Works

Wireshark is a free, open-source packet analyzer — also called a network sniffer or packet capture tool. It captures every packet that flows through a network interface on your machine and lets you inspect each one in full detail, down to the individual bit.

When you capture traffic, Wireshark puts your network adapter into promiscuous mode, meaning it captures ALL packets on the network segment — not just ones addressed to your machine. On a switched network you'll mostly see your own traffic, but on a hub or Wi-Fi in monitor mode you see everyone's.

▸ WHY THIS MATTERS FOR SOC ANALYSTS
Wireshark is the gold standard for network forensics. When an IDS/SIEM like Wazuh fires an alert, a SOC analyst pulls the raw packet capture (PCAP) to confirm what actually happened. No alert tells the full story — the packets do. Learning to read PCAPs is a core blue team skill.
1.1 — The Three Panels
PANEL LOCATION WHAT IT SHOWS
Packet List Top One row per captured packet. Shows time, source, destination, protocol, length, and a summary. Click any row to inspect it.
Packet Details Middle An expandable tree showing every protocol layer inside the selected packet. This is where you read the data.
Packet Bytes Bottom The raw bytes of the packet in hexadecimal on the left, ASCII text equivalent on the right.
1.2 — Capture vs Display Filters

These are two completely different things — beginners confuse them constantly:

TYPEWHEN APPLIEDSYNTAXEXAMPLE
Capture Filter BEFORE capture starts — limits what gets saved BPF (Berkeley Packet Filter) tcp port 80
Display Filter AFTER capture — hides/shows packets already captured Wireshark filter language tcp.flags.syn == 1
▸ PRO TIP
Always save your captures when doing lab attacks. File → Save As → .pcapng. This lets you re-analyze later and use the files in your LinkedIn posts or portfolio writeups.

The OSI Model in Wireshark

Every packet in Wireshark is a stack of protocol layers. When you click a packet and expand the details panel, each expandable section is one OSI layer. Data is encapsulated going down the stack, and decapsulated going up.

7ApplicationThe actual app data being sentHTTP · DNS · FTP · SSH
6PresentationEncoding, encryption, compressionTLS/SSL · JPEG · ASCII
5SessionManaging connections & sessionsNetBIOS · RPC · SOCKS
4TransportReliable delivery, ports, flow controlTCP · UDP · SCTP
3NetworkLogical addressing, routingIPv4 · IPv6 · ICMP
2Data LinkPhysical addressing, switchesEthernet · Wi-Fi · ARP
1PhysicalRaw bits on the wireFrame metadata
2.1 — Reading a Full Packet Stack

Here's what a single HTTP request looks like expanded in Wireshark, layer by layer:

▸ FRAME (LAYER 1)
Frame 42: 200 bytes on wire, captured on interface eth1
Metadata Wireshark adds — interface, timestamp, raw byte count. Nothing inside the actual network data.
▸ ETHERNET II (LAYER 2)
Destination: aa:bb:cc:dd:ee:ff    Source: 11:22:33:44:55:66    Type: IPv4 (0x0800)
MAC addresses only work on the LOCAL network segment. Routers strip and replace them at each hop.
▸ IPv4 (LAYER 3)
Source: 192.168.56.102 (Kali)    Destination: 192.168.56.101 (WEB01)    TTL: 64    Protocol: TCP
▸ TCP (LAYER 4)
Src Port: 54231    Dst Port: 80    Seq: 1    Flags: SYN
Ports identify WHICH APPLICATION on the machine gets the data.
▸ HTTP (LAYER 7)
GET /index.html HTTP/1.1    Host: 192.168.56.101
The actual request. Only visible if traffic is NOT encrypted (HTTP, not HTTPS).
CH · 03

Reading Hex Values & The Bytes Panel

The bottom panel in Wireshark shows the raw bytes of the packet in hexadecimal. This is the actual data that travelled on the wire. At first it looks like gibberish — but once you understand the structure, you can read any packet without the protocol parser.

3.1 — Hex Basics
DECIMALBINARYHEXWHAT IT MEANS
0000000000x00Null — zero
10000010100x0ANumber ten
15000011110x0FNumber fifteen
255111111110xFFMax value for 1 byte
▸ QUICK CONVERSION RULES
0x prefix means hexadecimal  ·  A=10, B=11, C=12, D=13, E=14, F=15
To convert hex to decimal: multiply left digit × 16, add right digit.
Example: 0x4F = (4×16) + 15 = 79
3.2 — Bytes Panel Layout

3.3 — Reading an IP from Hex
▸ HOW TO READ AN IPv4 ADDRESS FROM HEX
An IPv4 address is always exactly 4 bytes. Each byte = one octet of the IP.

Example: c0 a8 00 01
  c0 = 192    a8 = 168    00 = 0    01 = 1
Result: 192.168.0.1 — your typical home router IP

Every Protocol You Will Encounter

4.1 — Layer 2 Protocols
Ethernet II
FIELDSIZEWHAT IT CONTAINS
Destination MAC6 bytesMAC address of the receiver
Source MAC6 bytesMAC address of the sender
EtherType2 bytes0x0800=IPv4 · 0x0806=ARP · 0x86DD=IPv6
Payload46–1500 bytesThe actual data (IP packet, ARP, etc.)
DISPLAY FILTERS — ETHERNET
eth.addr == aa:bb:cc:dd:ee:ff// filter by MAC
eth.type == 0x0800// show only IPv4 frames
ARP — Address Resolution Protocol
ARP TYPEMEANINGSENDER
ARP RequestBroadcast: 'What MAC has this IP?'Any device needing to reach a local IP
ARP ReplyUnicast: 'That IP is mine, here is my MAC'The device that owns that IP
Gratuitous ARPDevice announces itself unprompted at bootDevice that just got an IP
ARP ProbeChecks if an IP is already in useDevice about to claim an IP (DHCP)
▸ ATTACK: ARP POISONING / SPOOFING
ARP has no authentication — anyone can send a fake ARP reply claiming to own any IP. Attacker sends fake ARPs saying "the gateway IP is at MY MAC" → All victim traffic routes through attacker (MITM).

⚠ MAN IN THE MIDDLE ⚠ ARP SPOOFING

Detection filter: arp.duplicate-address-detected
Lab test: use arpspoof on Kali to simulate, capture on Ethernet 2.
4.2 — Layer 3 Protocols
IPv4 Header Fields
FIELDSIZEWHAT IT DOES
Version4 bitsAlways 4 for IPv4
IHL4 bitsHeader length in 32-bit words. Value 5 = 20 bytes (minimum)
Total Length2 bytesTotal size of IP packet (header + data) in bytes
TTL1 byteHops remaining. Linux=64 · Windows=128 · Cisco=255
Protocol1 byteWhat's inside: 6=TCP · 17=UDP · 1=ICMP
Source IP4 bytesSender's IP address
Destination IP4 bytesReceiver's IP address
▸ TTL FINGERPRINTING
You can guess the OS of a sender by looking at the TTL value:
TTL ≈ 64 → Linux / macOS / Android
TTL ≈ 128 → Windows
TTL ≈ 255 → Cisco / network devices
The actual TTL you see is reduced by router hops — a Windows machine 3 hops away shows TTL=125.

4.3 — Layer 4: TCP & UDP
TCP Flags
SYN
Synchronize
Start a new connection. First packet of 3-way handshake.
ACK
Acknowledge
Acknowledging received data. Almost all packets after SYN.
FIN
Finish
Gracefully close connection. Normal termination.
RST
Reset
Abruptly kill connection. Port closed or firewall block.
PSH
Push
Send data immediately without buffering.
URG
Urgent
Urgent data in packet. Rare.
TCP 3-Way Handshake
STEPPACKETFLAGSMEANING
1Client → ServerSYNClient: I want to connect. My starting sequence is 0.
2Server → ClientSYN, ACKServer: OK. My sequence is 0. I confirm your byte 0.
3Client → ServerACKClient: Confirmed. Now let's send data.
DISPLAY FILTERS — TCP
tcp.flags.syn == 1 && tcp.flags.ack == 0// new connections / SYN scan
tcp.flags.reset == 1// connection resets — closed ports
tcp.flags.fin == 1// connection teardowns
tcp.stream eq 5// follow stream #5
tcp.analysis.retransmission// detect packet loss
4.4 — Layer 7: Application Protocols
DNS — Domain Name System
RECORD TYPEWHAT IT DOESEXAMPLE
AMaps hostname to IPv4 addressraghv.dev → 104.21.x.x
AAAAMaps hostname to IPv6 addressraghv.dev → 2606:...
CNAMEAlias — points to another hostnamewww → raghv.dev
MXMail server for a domainraghv.dev mail → mail.raghv.dev
PTRReverse lookup — IP to hostname192.168.1.1 → router.local
TXTText record — SPF, DKIM, verificationv=spf1 include:... -all
▸ ATTACK: DNS EXFILTRATION & TUNNELING
⚠ DNS EXFIL ⚠ DNS TUNNEL ⚠ CACHE POISON

Look for: very long query names, high-entropy random-looking subdomains, many queries to the same unusual domain.
Tools like dnscat2 and iodine use DNS as full C2 channel.
Detection: dns.qry.name.len > 50
HTTP Methods
METHODWHAT IT DOESSECURITY RELEVANCE
GETRetrieve a resourceParameters in URL — visible in Wireshark
POSTSend data to serverBody may contain login credentials
PUTUpload/replace a resourceWatch for large POST/PUT to unusual paths
DELETEDelete a resourceDestructive — flag in SOC context
OPTIONSWhat methods does server support?Reconnaissance — mapping server capabilities
▸ FINDING CREDENTIALS IN HTTP
Right-click any HTTP packet → Follow → HTTP Stream. You can read every HTTP header, URL, cookie, form field, and response body. This is how you find plaintext credentials in a capture.
TLS Handshake Messages
MESSAGEMEANING
Client HelloClient says: here are the TLS versions and cipher suites I support
Server HelloServer says: I choose this TLS version and cipher suite
CertificateServer sends its SSL certificate — includes domain name (verify this!)
Application DataEncrypted payload — you cannot read this without the private key
AlertError or close notification (e.g., certificate_unknown, bad_certificate)
▸ READING SNI IN ENCRYPTED TRAFFIC
In the Client Hello packet, expand: TLS → Handshake Protocol → Extension: server_name → Server Name

The SNI shows the domain name the client is connecting to, even in encrypted HTTPS traffic. Critical for SOC — you can identify which sites/services are accessed through TLS without decrypting anything.
Kerberos Attack Types
▸ KERBEROS ATTACKS IN ACTIVE DIRECTORY
⚠ KERBEROASTING ⚠ AS-REP ROASTING ⚠ PASS THE TICKET ⚠ GOLDEN TICKET

Kerberoasting: attacker requests service tickets for service accounts, cracks them offline. Filter: kerberos.msg_type == 30 from unusual sources.
Golden Ticket: forging a TGT using the KRBTGT hash — gives permanent domain admin access.

Home Lab Attack Scenarios

These are the captures you should be doing in your lab (Kali → WEB01/DC01 on Ethernet 2 / 192.168.56.x) to build your skills and LinkedIn portfolio.

🔍 SCENARIO 7.1 — NMAP PORT SCAN DETECTION
STEP 01Start Wireshark on Ethernet 2Select the 192.168.56.1 interface on Windows (or eth1 on Kali)
STEP 02Run SYN scan from Kalinmap -sS 192.168.56.101

STEP 04Apply filterarp — watch for duplicate ARP replies
STEP 05Check for MITMTraffic from WEB01 now routes through Kali before reaching gateway
